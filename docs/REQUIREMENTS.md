# Eating Out - 要件定義書

## 1. 概要

### 1.1 アプリの目的

外食に特化した食事管理アプリ。従来アプリの「写真撮影→カロリー推定」の手間を排除し、**価格・カロリーが明確な外食**の特性を活かして、希望予算・カロリー内のメニューを地図と連携して提案する。

### 1.2 解決する課題

| 課題 | 従来アプリ | 本アプリ |
|------|-----------|----------|
| カロリー入力 | 写真撮影や手入力が面倒 | チェーン店はメニュー選択のみ |
| 予算管理 | 別アプリで管理 | 希望金額でフィルタ・提案 |
| 店舗検索 | 地図と食事管理が分離 | 位置×予算×カロリーで一括提案 |

### 1.3 ターゲットユーザー

- 外食頻度が高い（週2回以上）
- カロリー・予算を意識したい
- 主にチェーン店を利用する層

## 2. 機能要件

### 2.1 コア機能

| 機能ID | 機能名 | 説明 | 優先度 |
|--------|--------|------|--------|
| F001 | ユーザー設定 | 1日の目標カロリー・予算を設定 | 必須 |
| F002 | 店舗検索 | 現在地から半径指定で飲食店を検索 | 必須 |
| F003 | メニュー提案 | 残り予算・残りカロリー内のメニューを提案 | 必須 |
| F004 | 食事記録 | 食べたメニューを記録（カロリー・金額を自動反映） | 必須 |
| F005 | 残高表示 | 本日の残りカロリー・残り予算を表示 | 必須 |

### 2.2 拡張機能（フェーズ2以降）

| 機能ID | 機能名 | 説明 | 優先度 |
|--------|--------|------|--------|

## 3. 外部API選定

### 3.1 採用API

| API | 用途 | 選定理由 |
|-----|------|----------|
| **Google Maps Platform (Places API)** | 店舗検索・位置情報・営業時間 | 網羅性・正確性が最高。営業中フィルタに必須 |
| **Geolocation API** | 現在地取得 | スマホのGPSと連携 |

### 3.2 補完API（検討）

| API | 用途 | 注意点 |
|-----|------|--------|
| ホットペッパーグルメ | ジャンル・クーポン | メニュー・カロリーは非対応。予約連携に有用 |
| ぐるなび | 店舗詳細 | 同上。国内店舗の補完用途 |

### 3.3 メニュー・カロリーAPIの課題（重要）

**現状、日本の全チェーン店のメニュー・価格・カロリーを提供するAPIは存在しない。**

→ **自社DB構築が競合差別化の要**となる。

| データ取得方法 | メリット | デメリット |
|----------------|----------|------------|
| 公式サイトの栄養成分表 | 正確・法的リスク低 | 手動/半自動更新が必要 |
| Webスクレイピング | 自動化可能 | 利用規約・法的リスク要確認 |
| ユーザー投稿 | カバー範囲拡大 | 精度管理・モデレーションが必要 |

**推奨**: 大手チェーン（大戸屋、すき家、サイゼリヤ、マクドナルド等）に絞り、公式公開データを定期更新。フェーズ2でユーザー投稿を検討。

## 4. データベース設計

### 4.1 ER概要

```
Users ──< Meal_Logs >── Menus
                │
                └── Restaurants (chain_id経由でChainsに紐づく)

Chains ──< Restaurants
    │
    └──< Menus
```

### 4.2 テーブル定義

#### Users（ユーザー）

| カラム | 型 | 説明 |
|--------|-----|------|
| user_id | PK | ユーザーID |
| daily_calorie_limit | INT | 1日の目標カロリー（kcal） |
| daily_budget_limit | INT | 1日の目標外食費（円） |
| created_at | TIMESTAMP | 登録日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### Chains（チェーン店）

| カラム | 型 | 説明 |
|--------|-----|------|
| chain_id | PK | チェーンID |
| chain_name | VARCHAR | チェーン名（例: 大戸屋） |
| official_url | VARCHAR | 栄養成分表URL（更新用） |
| created_at | TIMESTAMP | 登録日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### Restaurants（実店舗）

| カラム | 型 | 説明 |
|--------|-----|------|
| restaurant_id | PK | 店舗ID |
| chain_id | FK | 属するチェーンID |
| name | VARCHAR | 店舗名（例: 大戸屋 八王子店） |
| latitude | DECIMAL | 緯度 |
| longitude | DECIMAL | 経度 |
| place_id | VARCHAR | Google Places APIのplace_id（重複防止・更新用） |
| created_at | TIMESTAMP | 登録日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### Menus（メニュー）

| カラム | 型 | 説明 |
|--------|-----|------|
| menu_id | PK | メニューID |
| chain_id | FK | 提供チェーンID |
| menu_name | VARCHAR | メニュー名 |
| price | INT | 価格（円） |
| calories | INT | カロリー（kcal） |
| is_available | BOOLEAN | 販売中フラグ（メニュー改定対応） |
| created_at | TIMESTAMP | 登録日時 |
| updated_at | TIMESTAMP | 更新日時 |

#### Meal_Logs（食事記録）

| カラム | 型 | 説明 |
|--------|-----|------|
| log_id | PK | 記録ID |
| user_id | FK | ユーザーID |
| menu_id | FK | メニューID（NULL可：非チェーン時） |
| eaten_at | TIMESTAMP | 食べた日時 |
| manual_calories | INT | 手動入力カロリー（非チェーン時） |
| manual_price | INT | 手動入力金額（非チェーン時） |
| created_at | TIMESTAMP | 登録日時 |

### 4.3 Gemini案との主な差分

- **place_id**: Google APIとの紐づけ・重複防止用に追加
- **is_available**: メニュー改定時の論理削除対応
- **manual_calories / manual_price**: 非チェーン店の記録対応（拡張性）
- **official_url**: チェーンの栄養成分表URL（更新自動化の足がかり）

## 5. 処理フロー

### 5.1 メインシナリオ：夜ご飯を探す

```
1. ユーザーが「夜ご飯を探す」をタップ
2. 現在地をGPSで取得
3. Google Places API で「半径1km以内の飲食店」を取得
4. 取得店舗と自社Restaurantsを照合（place_id or 名前＋位置でマッチ）
5. マッチしたチェーンのMenusを取得
6. ユーザーの「本日残り予算」「本日残りカロリー」でフィルタ
7. 距離順でソートし、提案リストを表示
   → 「徒歩3分の大戸屋で、さばの炭火焼き定食（990円・672kcal）」
```

### 5.2 残り予算・カロリーの計算

- **リセット**: 日付変更（0時）でリセット
- **残り** = 目標 - 本日のMeal_Logs合計

## 6. 非機能要件

### 6.1 パフォーマンス

- 店舗検索〜提案表示: 3秒以内
- Places APIのレスポンスをキャッシュ（同一エリア・一定時間）

### 6.2 コスト

- Google Maps Platform: 従量課金のため、キャッシュ・リクエスト最適化が重要
- 無料枠の活用を前提に設計

### 6.3 セキュリティ

- APIキーは環境変数で管理、リポジトリに含めない
- ユーザーデータは個人を特定しない形で保持

## 9. リスクと対策

| リスク | 対策 |
|--------|------|
| メニューデータの更新負荷 | チェーン数に絞る。更新頻度の低いチェーンを優先 |
| Places APIコスト | キャッシュ、検索半径の最適化、無料枠の監視 |
| 非チェーン店のカバー不足 | フェーズ2で「簡易記録」機能を追加し、ユーザー体験を維持 |
| スクレイピングの法的リスク | 公式公開データのみ使用。利用規約を確認 |
